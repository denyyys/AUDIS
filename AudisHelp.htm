Overview

AUDIS is a Windows desktop SIP telephony server and client built on the .NET 8 platform. It can:

  • Answer incoming SIP calls and play audio menus (DTMF-driven IVR)
  • Record calls and voicemails
  • Respond to caller questions using an on-device AI (Whisper + Ollama)
  • Make outbound calls via the built-in SIP Client
  • Report the current time and weather

Quick Start

  1. Set your Public IP in the Config tab to your machine's LAN IP (e.g. 192.168.100.64).
  2. Place your WAV audio files (8 kHz, mono, 16-bit PCM) in the audio\ folder.
  3. Click Start Service — AUDIS listens on UDP port 5060.
  4. Point any SIP phone or PBX extension at sip:audis@<YOUR-IP>:5060.
  5. Call it. You will hear eliska.wav followed by the DTMF menu.
WAV format requirement: All audio files must be 8 000 Hz, mono, 16-bit PCM. Convert with: ffmpeg -i input.mp3 -ar 8000 -ac 1 -acodec pcm_s16le output.wav

Key Mappings

When a call is connected, callers press DTMF digits to hear different audio. Each digit maps to an audio file name or a system action.

KeyDefault ActionNotes
0eliska.wavGreeting / main menu
1–5cibula.wav, sergei.wav…Custom audio files
6SYSTEM_STATUS Speaks the number of active calls
7INFO_PACKAGE Speaks current time and temperature
8VOICEMAIL Records a voicemail; saved to voicemail\
*AI_START Activates AI transcription and response mode
#AI_STOP Stops AI recording and processes the captured speech
You can change the value in the Key Mappings tab. Use any filename from the audio\ folder, or one of the system keywords above.

AI Mode (*)

When a caller presses *, AUDIS enters AI mode:

  1. Caller speaks their question or message.
  2. Caller presses # to stop recording.
  3. The recorded audio is transcribed by Whisper.net (ggml-base.bin).
  4. The transcribed text is sent to Ollama (local LLM).
  5. The AI response is converted to speech via Google TTS and played back.

Prerequisites

  • Ollama — ollama.ai — install and run ollama pull gemma3:1b
  • Whisper model — download ggml-base.bin from Hugging Face and place it next to AudisService.exe
  • ffmpeg — must be in your system PATH (for TTS audio conversion)
  • Internet access — Google TTS requires internet for voice synthesis

Codec auto-detection

AUDIS automatically detects whether the caller's phone sends PCMU (μ-law, payload type 0) or PCMA (A-law, payload type 8) and decodes accordingly before feeding to Whisper. If you see the fallback response "Kdo jsi?" every time, check the console log for [WHISPER] Silence check output.

SIP Client

The SIP Client lets AUDIS make outbound calls from the same machine. It runs on a separate SIP transport (port 5061) so it does not interfere with the incoming call server on port 5060.

Making a call

  1. Click the SIP Client toolbar button or open it from the toolbar.
  2. Type a SIP address in the top field, for example:
    • 192.168.1.5 — calls port 5060 on that IP
    • sip:200@pbx.local — calls extension 200 at pbx.local
    • sip:alice@example.com:5080 — custom port
  3. Press Call or hit Enter.

During a call

Once connected, AUDIS plays eliska.wav to the remote party and waits for them to press DTMF digits, exactly as if they had called the server. You can also press the on-screen dialpad buttons during a call to send DTMF to the remote end.

Logs

AUDIS maintains two separate logs:

  • System Logs tab — server activity: incoming calls, DTMF, audio playback, errors
  • SIP Client window log — outbound call activity, separate log file on export

Exporting logs

Use the Export Log toolbar button to instantly export and open the server log. Inside the SIP Client window, click Export to save the client log. Both save timestamped files in logs\.

Always export logs before closing AUDIS if you need to investigate an issue — the in-memory log is lost on exit.

Audio Files

All audio files live in <install dir>\audio\.

Required format: WAV, 8 000 Hz, mono, 16-bit PCM (signed). Files in any other format will play as noise or silence. 
ffmpeg -i input.mp3   -ar 8000 -ac 1 -acodec pcm_s16le output.wav
ffmpeg -i input.ogg   -ar 8000 -ac 1 -acodec pcm_s16le output.wav
ffmpeg -i input.m4a   -ar 8000 -ac 1 -acodec pcm_s16le output.wav

Use the Files tab to browse the audio folder and verify which files are present. Click Open Folder to open it in Explorer.

Troubleshooting

Caller hears nothing / call drops immediately

  • Verify eliska.wav exists in the audio\ folder
  • Check it is 8 kHz mono 16-bit PCM (not MP3 or stereo)
  • Check Public IP matches your machine's actual LAN IP
  • Open UDP port 5060 (SIP) and ports 10000–20000 (RTP) in Windows Firewall

DTMF not working

  • Check the System Logs for [RTP DEBUG] lines — they show what payload types arrive
  • If you see payload types 96–127 but no detection, the phone may send end-of-event only once; check debounce logs
  • Try switching from RFC 2833 to SIP INFO (or vice-versa) in your SIP phone's settings
  • Some phones require in-band DTMF; AUDIS currently supports RFC 4733 and SIP INFO only

AI returns "Kdo jsi?" every time

  • Check that ggml-base.bin is present next to the EXE
  • Look for [WHISPER] Silence check: xx/100 bytes are silence — if >90, audio is not reaching the buffer
  • Check codec: [RTP AUDIO] Codec detected: PayloadType=X — should be 0 (PCMU) or 8 (PCMA)
  • Ensure the caller speaks for at least 2 seconds before pressing #

Ollama AI not starting

  • Run ollama serve in a terminal manually to see error output
  • Run ollama pull gemma3:1b to download the model
  • Check port 11434 is not used by another process: netstat -ano | findstr 11434

SIP Client cannot connect

  • Check that the target SIP address is reachable (ping the IP first)
  • Ensure port 5061 is open in Windows Firewall
  • Verify the Public IP in SIP Client Settings matches your LAN IP
  • If calling through a SIP proxy, enable registration and fill in the account details
